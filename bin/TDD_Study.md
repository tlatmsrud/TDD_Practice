# 자바지기 박재성님의 우아한 테크 영상을 정리한 내용입니다.
# 1. TDD(Test Driven Development)란?
- 테스트 중심의 개발 = 테스트 코드가 중심이 되는 개발
- 개발 방법론 중 하나

# 2. 현업에서의 TDD
1. 개발 단계에서 시간이 오래 걸림
2. TDD가 모든 버그를 잡아주진 않음

# 3. TDD를 사용해야하는 이유
1. 개발 초기단계에 설계 오류를 찾을 수 있음
    - 테스트 코드의 첫 시작은 기능에 대한 실패, 성공에 대한 케이스를 먼저 나열하는 것이며, 이를 위해서는 당연히 설계를 더 자세히 볼수밖에 없음.

2. 상세한 개발 진행도를 확인할 수 있음  
    - 기능별로 구현된 테스트코드를 실행시키는 것만으로 전체적인 개발 진행도 확인이 가능함.
  
3. 오버 엔지니어링 방지
   - 테스트 케이스가 모두 성공하면 다음 작업으로 넘어갈 수 있는 지표가 됨.

4. 유지보수 비용 감소
    - 로직 수정 시 영향도 체크가 쉽고, 기능별로 개발되었기에 로직 수정도 원활함.

# 4. TDD 연습
1. String이나 Collection 등의 API 테스트 코드 작성을 통해 JUnit 기본 문법 익히기.
2. 구현하는 함수 중 Input과 Output이 명확한 클래스 메소드에 대한 테스트 코드 작성 연습.

# 5. 리펙토링 연습
1. 테스트 코드는 변경하지 말고 테스트 대상 코드를 개선하는 연습.

# 6. 리펙토링 규칙
1. 한 메서드에 오직 한 단계의 들여쓰기만 한다.
2. 함수의 최대길이는 15줄로 제한.
3. Else, Switch case문 사용 제한.
4. 함수는 한가지 기능에 집중하도록 '극단적'으로 제한.
5. 로컬변수가 필요한지 의심. 다른 곳에서 사용할 일이 없다면 계산된 값을 넘겨주기.
6. compose method 패턴을 적용.
7. 한번에 모든 규칙을 지키려 하지말고 한번에 한가지씩 명확하고 구체적인 목표를 갖고 연습.
8. 일급 콜렉션을 사용. - 콜렉션을 클래스로 래핑.
9. 3개 이상의 인스턴스 변수를 가진 클래스는 사용 제한.

# 7. 리펙토링 연습하기 좋은 프로그램 요구사항
1. 게임과 같이 요구사항이 명확한 프로그램
2. 모바일, 웹, DB와의 의존관계가 없는 프로그램
3. 약간은 복잡한 로직이 있는 프로그램
4. 예) 로또, 사다리타기, 체스, 지뢰찾기(UI는 콘솔)

# 8. 객체지향 생활체조 규칙
1. 한 메서드에 오직 한 단계의 들여쓰기만 허용하기
2. else 사용 제한하기
3. 모든 원시값과 문자열을 포장하기
4. 한 줄에 점을 하나만 찍기
5. 줄여쓰지 않기
6. 모든 엔티티를 작게 유지하기
7. 3개 이상의 인스턴스 변수를 가진 클래스를 사용하지 않기
8. 일급 콜렉션 사용하기
9. 게터, 세터, 프로퍼티를 사용하지 않기.
